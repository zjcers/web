Linter-GCC*

*(Fork of)

Origins

If there's an Atom package that I immediately search for when setting up a new installation, it's linter-gcc. I love it. On the other hand, it has some warts. I found a bug in it while working on my C-library. Making a freestandingC project wasn't what pushed linter-gcc to the limits. Rather, it was the fact that I like to have more than one project open at a time.

The original branch of linter-gcc is hardcoded to only search its configuration file in the directory tree of the first open project in Atom. I had originally set out to quickly correct this and send a pull request to the author. That sounded easy at first. Once I started reading through the codebase, though, everything changed. I found where the relevant incorrect assumptions were made, but changing them caused an error it really shouldn't have in another method.

Finding this, I decided to go down the rabbit hole. I tried to grok my way through the code, but, and I hate to come off as overly-critical, it was way too hard to read. Some parts of it I simple could not understand, and some parts were...inefficent. So I decided to refactor a bit of it. Well, one thing led to another, and I spent a week's worth of free time rewriting nearly the whole project. Time well spent, I think.

Status

Current Status (versus the original)

  • I fixed the bug I set out to (multiple projects open at once)
  • I regressed in functionality by removing the lint-on-the-fly support
  • Everything is almost-completely rewritten, except for config.js
  • About 4 fewer dependencies

Future plans
