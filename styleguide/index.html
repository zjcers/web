int atoi(const char* src) {
	size_t i = 0;
	int ret = 0;
	bool inNumber = false;
	bool isNegative = false;
	for(; src[i] != 0; i++) {
		/*before the number is found in the string*/
		if (inNumber == false) {
			/*check if there is a leading - sign*/
			if (src[i] == '-') {
				inNumber = true;
				isNegative = true;
			}
			/*if not, check if the leading digit has been found*/
			else if (src[i] >= '0' && src[i] <= '9') {
				inNumber = true;
				ret = addToNumber(ret, src[i]);
			}
		}
		/*inside the number, check if the current char is still a digit*/
		else if (src[i] >= '0' && src[i] <= '9'){
			ret = addToNumber(ret, src[i]);
		}
		/*if the current char is not a digit, the number is over*/
		else {
			break;
		}
	}
	if (isNegative) {
		return ret*-1;
	}
	return ret;
}

Comments

I always use the "/**/" comment format, even for single-line comments. I believe this encourages me to put comments on their own lines. In turn, I think this leads to more meaningful content as opposed to quick comments at the end of a line. It should be noted that I'm a bit lazy and don't use the traditional leading asterisks like I should. This is for consistency among the other languages that I write in that don't use that convention.

Indentation and bracket placement

I always use hard tabs when possible in any language. I know there exists a lot of controversey on this regard, and I don't have a zealous opinion on this. On the other hand, hard tabs are easier to navigate when the text editor you're using doesn't know how to detect soft tabs and skip them automatically. Also, I have yet to meet a language that honestly refuses them, though I know M4 refuses to parse soft tabs.

I put my opening brackets on the same line as the expression they belong to. Some people like it the other way. That's fine, but this way is easier for me to read. It allows me to look straight up from the respective closing bracket and see the exact statement it belongs to rather than having to skip up an extra line.

Naming Stuff

As per C tradition, I name public-facing things underscore-deliminated style. Private things such as locals and helper functions are camelCased. This isn't very C-like, but I like it. It makes it easier for me to tell when something is intended to be global or not.

Functions and other syntax

I always omit the space between the function name and the argument list in any language that supports it. I have no reasoning to back this up other than it saves my thumb a trip to the space bar. This is pretty much the only place I omit whitespace. Argument lists either in declarations or calls are always space-separated. I think it looks a lot cleaner.

Return is not a function. I don't treat as so (though I did when I first learned C++). sizeof is also not a function, though I'm with Linus on this one. It makes more sense to treat it as one.

C++

I use the same styling rules as C for the most part. I'll update this section once I get back into a C++ project again.

Python

PEP8 says that I should use 4-space tabs. It says a lot of other things, too. A lot of those I don't care to read or just ignore. Like the recommendation to use underscore- deliminated variable names. I like camelCase too much for that. It looks better.

I try to write JavaScript like I write C/C++ as much as possible. The comment and indentation styles I use are mostly the same. I like the new ES6 variable statements (let and const). They make scoping a lot easier to track. On the other hand, the new short function statements (<varname> =>...) aren't very readible and I avoid using them.
